# 2. Progressive Blake3 Hashing

A primary vector of P2P disruption is chunk poisoning, where a malicious node injects corrupted bytes to stall downstream decoders. To neutralize this without introducing verification latency, the segment payload is divided into $16\text{ KB}$ block symbols and structured as a **Blake3 Merkle Tree**:

```text
                                [ Signed Manifest Root ]  (Ed25519 Signed by Source)
                                      /          \
                                 [Hash L]      [Hash R]
                                  /    \        /    \
                               H(0)   H(1)    H(2)   H(3)
                                |      |       |      |
                              [B_0]  [B_1]   [B_2]  [B_3] <-- 16KB Data Blocks
```

Let a segment be divided into $N$ blocks: $\mathcal{B} = \{B_0, B_1, \dots, B_{N-1}\}$. The Merkle Tree is constructed as follows:
1.  **Leaf Hashing:** Every block $B_i$ is hashed using the ultra-fast Blake3 hash:
    $$h_i = \text{Blake3}(B_i)$$
2.  **Internal Node Hashing:** Parent nodes are generated by hashing the concatenation of their children:
    $$h_{x,y} = \text{Blake3}(h_x \parallel h_y)$$
3.  **Root Generation:** The process repeats until a single 32-byte Root Hash $H_{\text{root}}$ is obtained.
4.  **Manifest Signature:** The broadcaster signs the Root Hash along with the segment sequence number:
    $$\text{Sig}_{\text{Manifest}} = \text{Sign}_{SK_{\text{Source}}}(\text{SegmentID} \parallel H_{\text{root}})$$

When a client downloads a single block $B_i$, the sender bundles it with the **Merkle Path Proof** (the set of sister hashes needed to reconstruct the path to the root). For example, to verify $B_0$, the client receives $B_0$ and the sister hashes $\{h_1, h_{2,3}\}$.

The client executes the progressive verification:
$$h_0' = \text{Blake3}(B_0)$$
$$h_{0,1}' = \text{Blake3}(h_0' \parallel h_1)$$
$$H_{\text{root}}' = \text{Blake3}(h_{0,1}' \parallel h_{2,3})$$

If $H_{\text{root}}' == H_{\text{root}}$ (extracted from the signed manifest), **the client validates the block and immediately forwards it to downstream children**, achieving sub-millisecond, zero-trust progressive verification.
