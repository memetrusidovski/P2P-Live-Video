# 2. The Propagation Latency Model

Live video streaming is fundamentally constrained by a **Playout Deadline** $\Delta_{\text{playout}}$. If a chunk $C_k$ is generated by the broadcaster at time $T_k$, it must be fully assembled and validated in the playback buffer of every peer $i$ before time $T_k + \Delta_{\text{playout}}$. For low-latency interactive streaming (such as sports, esports, or live chats), $\Delta_{\text{playout}}$ is constrained to:
$$\Delta_{\text{playout}} \le 3.0 \text{ to } 5.0\text{ seconds}$$

In a peer-to-peer overlay, the propagation delay of a chunk is a function of the path length (hop count) from the source to the node. Let $h_i$ be the hop distance of peer $i$ from the source. The time $T_{\text{recv}}(i, C_k)$ at which peer $i$ receives chunk $C_k$ is mathematically modeled as:
$$T_{\text{recv}}(i, C_k) = T_k + \sum_{j=1}^{h_i} \left( \tau_{\text{trans}}(j) + \tau_{\text{prop}}(j) + \tau_{\text{queue}}(j) \right)$$

where:
*   $\tau_{\text{trans}}(j) = \frac{\text{BlockSize}}{u_{\text{parent}}(j)}$ is the transmission delay of forwarding the chunk's constituent blocks.
*   $\tau_{\text{prop}}(j) = RTT(j, j-1)/2$ is the physical propagation delay (fiber latency) between the node and its parent.
*   $\tau_{\text{queue}}(j)$ is the processing and scheduling queue delay in the node's buffer.

To ensure no playback interruption occurs, the protocol must satisfy:
$$\max_{i \in \text{Swarm}} \left( T_{\text{recv}}(i, C_k) - T_k \right) < \Delta_{\text{playout}}$$
