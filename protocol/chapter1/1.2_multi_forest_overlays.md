# 1.2 Mathematical Formulation of Dynamic Multi-Forest Overlays

## 1.2.1 The Multi-Forest Concept

To prevent network interface saturation and optimally leverage asymmetric home internet connections, our protocol divides the live video stream into $M$ independent, orthogonal sub-streams (or "slices"). 

Let the primary video feed be encoded with H.264/SVC (Scalable Video Coding) or divided into distinct frame streams using Multiple Description Coding (MDC). If the primary video bitrate is $B$, each sub-stream $m \in \{1, 2, \dots, m\}$ has a bitrate of:
$$B_m = \frac{B}{M}$$

For example, a $6\text{ Mbps}$ stream divided into $M = 6$ sub-streams requires a small transmission capacity of $1\text{ Mbps}$ per sub-stream. 

Instead of forcing all peers into a single tree, the protocol builds a **Multi-Forest** overlay $\mathcal{F}$:
$$\mathcal{F} = \{T_1, T_2, \dots, T_M\}$$

where each $T_m$ is an independent, self-organizing distribution tree dedicated to routing the symbols of sub-stream $m$.

---

## 1.2.2 The Orthogonal Placement Rule

The critical constraint of the Multi-Forest overlay is the **Orthogonal Placement Rule**, designed to balance the upload burden across different sets of physical nodes. 

Let $\mathcal{N}$ represent the set of all active peers in the swarm. For any peer $i \in \mathcal{N}$, let $\text{role}(i, T_m)$ represent the role of node $i$ in the distribution tree $T_m$, defined as:
$$\text{role}(i, T_m) \in \{\text{Source}, \text{Relay}, \text{Leaf}\}$$

To prevent any single peer from saturating its upload bandwidth, a peer $i$ is allowed to act as an active uploader (Relay) in at most **one** tree $T_a$, and must act as a receiver-only node (Leaf) in all other trees $T_b$ where $b \ne a$:

$$\forall i \in \mathcal{N}, \quad \sum_{m=1}^{M} \mathbb{I}(\text{role}(i, T_m) = \text{Relay}) \le 1$$

where $\mathbb{I}(\cdot)$ is the indicator function.

```text
       Sub-Stream 1 Tree (T1)                 Sub-Stream 2 Tree (T2)
         [ Source (Root) ]                      [ Source (Root) ]
           /           \                          /           \
     [ Peer A ]     [ Peer B ]              [ Peer C ]     [ Peer D ]   <-- Relays
      /      \       /      \                /      \       /      \
  [Peer C] [Peer D] [Peer E] [...]       [Peer A] [Peer B] [Peer E] [...] <-- Leaves
```

*   **Peer A** is an active **Relay** in $T_1$, forwarding Sub-Stream 1 to downstream nodes. It is a **Leaf** in $T_2$, receiving Sub-Stream 2 from Peer C.
*   **Peer C** is an active **Relay** in $T_2$, forwarding Sub-Stream 2 to downstream nodes. It is a **Leaf** in $T_1$, receiving Sub-Stream 1 from Peer A.

This architecture ensures that a peer with $10\text{ Mbps}$ upload capacity can act as a high-tier relay for one sub-stream ($1\text{ Mbps}$ in, $8\text{ Mbps}$ out), while still receiving all other sub-streams as a leaf, utilizing its asymmetric $50\text{ Mbps}$ download line.

## 1.2.3 Parent Selection and Tree Join Algorithm

When a new peer $i$ joins the network, it must establish parent connections for all $M$ sub-streams. It executes the following deterministic algorithm:

```text
Algorithm 1: Multi-Forest Tree Join Protocol
Input: Joining Peer 'i', Target StreamID, Available Slices M
Output: Parent Set P_i = {p_i(1), p_i(2), ..., p_i(M)}

1:  P_i <- empty set
2:  Calculate local peer score Score(i) based on historical upload statistics
3:  Identify primary slice 'a' to relay: 
        a <- Score(i) % M  (Slices are allocated modulo-wise to balance forest)
4:  
5:  For each slice m from 1 to M do:
6:      If m == a and Score(i) > High_Threshold then:
7:          // Join as a potential Relay node in Tree T_a
8:          Query DHT for active high-tier nodes in T_a
9:          Select parent candidate 'p' with lowest RTT and hop-count
10:         Send RELAY_JOIN_REQUEST(i, slice=a) to 'p'
11:         If accepted then:
12:             p_i(a) <- p
13:             Mark role(i, T_a) <- Relay
14:         Else:
15:             Fall back to Leaf join
16:     
17:     If p_i(m) is empty then:
18:         // Join as a Leaf node in Tree T_m
19:         Query local Passive Set for peers participating in T_m
20:         Select parent candidate 'p' with verified spare capacity
21:         Send LEAF_JOIN_REQUEST(i, slice=m) to 'p'
22:         p_i(m) <- p
23:         Mark role(i, T_m) <- Leaf
24: 
25: Return P_i
```

## 1.2.4 Localized Topology Healing (Self-Optimization)

Trees are inherently vulnerable to intermediate node departures. When a parent $p_i(m)$ in tree $T_m$ disconnects or experiences a packet-loss spike, the downstream children must heal the tree locally without invoking global restarts:
1.  **Direct Promotion:** The parent's child with the highest local contribution score (the "deputy child") is immediately promoted to take the failed parent's place.
2.  **Sibling Re-attachment:** The remaining sibling nodes re-attach to this newly promoted parent, restoring the subtree branch in less than $300\text{ ms}$.
3.  **Active Depth Balancing:** If a child's hop distance $d_i(m)$ exceeds the target limit $D_{\text{max}}$, the child queries the gossip layer for higher-layer nodes with idle capacity and executes a parent-switch operation to migrate upward.
